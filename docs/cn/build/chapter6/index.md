# 第六章 多渠道接入：让每个平台都发挥特长

> **核心提示**：我们已经知道 OpenClaw 通过 Gateway 接收消息、通过队列调度任务。但真正的魔法在于——同一个 Agent，如何在 Telegram、Discord、飞书这些截然不同的平台上，既保持一致的体验，又能发挥每个平台的特长？

---

## 1. 不只是"能收到消息"那么简单

在第二章，我们看到了Gateway的架构图——它像一个大门口，各种平台的消息从这里进入系统。但如果我们只把渠道当成"消息的入口"，就浪费了一个巨大的机会：**每个平台都有自己的超能力**。

想象你有一个助理。你在微信里问他问题时，他只能发文字；你在钉钉里问他时，他却可以用卡片展示任务列表、用按钮让你确认操作。这不是助理变了，而是**平台的能力被充分利用了**。

OpenClaw的多渠道架构，就是为了让每个平台都能"各显神通"——不同平台有各自独特的消息特性（如富文本、卡片消息、线程等）。

但同时，无论用户在哪个平台，Agent的**核心能力**和**记忆**都是一致的。这就需要一个精巧的设计——既能统一核心，又能发挥个性。

## 2. 翻译官也有"基本法"

还记得我们用"翻译官"来比喻适配器吗？实际上，这个翻译官不是随便发挥的，它有一套严格的工作规范。

### 2.1 Gateway：统一的消息处理中心

OpenClaw的**Gateway**是渠道适配器的运行时管理器，负责启动、停止和管理各平台连接。每个渠道通过实现`ChannelPlugin`接口来接入系统，包含多个专门的适配器（如`outbound`、`config`、`gateway`等）。

```
Telegram适配器 ──┐
Discord适配器 ───┼──> Gateway <──> Agent核心
飞书适配器 ──────┘
```

适配器们不需要知道其他适配器的存在，它们各自负责：

1. **接收平台消息**：通过`ChannelGatewayAdapter`启动平台监听，在渠道内部处理消息
2. **发送回复**：通过`ChannelOutboundAdapter.sendText()`等方法直接发送到平台
3. **配置能力**：通过`ChannelConfigAdapter`和`capabilities`配置定义平台功能

这个设计的巧妙之处在于**解耦**。你想加一个Matrix平台？实现Matrix的`ChannelPlugin`接口即可，Agent核心一行代码都不用改。

### 2.2 标准消息格式：翻译的"词典"

各渠道适配器与系统核心之间传递的消息，必须遵循统一格式。这就像大家约定好的一套"词典"：

**入站消息（用户发来的）**包含：
- **谁发的**：发送者ID（`from`）、账户ID（`accountId`）
- **发到哪儿**：聊天类型（`chatType`，私聊还是群聊？）、聊天ID（`chatId`）
- **内容**：消息正文（`body`），文字、图片、还是文件？
- **上下文**：引用哪条消息（`quote`）、发送时间等元信息

**出站消息（Agent回复的）**包含：
- **发给谁**：目标房间/对话ID（`to`）
- **内容**：通常是Markdown格式（`text`）
- **附件**：要发的图片或文件（`mediaUrl`）

注意一个关键点：**出站消息的内容通常是Markdown**。为什么？因为Markdown是通用格式，几乎所有平台都支持（只是支持程度不同）。翻译官拿到Markdown后，根据平台能力决定是否转换。比如Telegram使用专门的`chunkMarkdownText`函数进行Markdown分块处理，而Discord使用默认的文本处理方式。具体支持程度请参考各平台官方文档。

这就好比你让助理发一封邮件。助理自己会判断：如果是正式邮件用Word格式，如果是即时通讯用简洁文字。

## 3. 能力接口：让每个平台"各显神通"

现在来看精彩的部分——**不是所有平台都一个样**。OpenClaw怎么知道Discord支持Embed而Telegram不支持？答案是**平台能力配置**。

每个渠道在配置时，都可以设置其**能力选项**。这不是简单的"是/否"，而是一套可配置的功能开关：

### 3.1 平台能力配置

OpenClaw通过配置项来控制各平台的能力：

```
基础能力（所有平台都支持）：
- 接收消息 ✓
- 发送消息 ✓

可选能力（部分平台支持）：
- 发送"正在输入..."提示（typingIndicator）
- 发送图片/文件（mediaMaxMb 控制大小限制）
- 给消息加表情反应（部分平台支持，如Discord的ackReaction、Telegram的reactionLevel）
- 编辑已发送的消息（平台支持情况下）
- 流式消息预览（streaming，不同平台实现方式不同）
- 通过Webhook接收消息（webhookUrl 配置）
```

以Discord为例，其能力配置包括：
```json5
{
  channels: {
    discord: {
      enabled: true,
      ackReaction: "👀",  // 处理消息时发送表情反应
      streaming: {       // 流式消息配置
        blockStreamingCoalesceDefaults: { /* ... */ }
      },
      actions: {
        messages: true,     // 发送消息
        reactions: true,    // 表情反应
        threads: true,      // 话题线程
      }
    }
  }
}
```

而飞书的配置示例：
```json5
{
  channels: {
    feishu: {
      enabled: true,
      typingIndicator: true,  // 飞书支持typing reaction，设为false可禁用
      // 飞书使用卡片消息和Markdown渲染，配置结构与其他平台有所不同
    }
  }
}
```

### 3.2 流式预览：一个好玩的例子

来看**streaming**能力的实际应用：

你在Discord问Agent："帮我分析这个项目"——这需要几秒钟。如果什么都不显示，你会怀疑是不是消息没发出去。

所以Discord翻译官可以开启流式预览：先发一个预览消息"🤔 让我想想..."，然后随着Agent生成内容，不断编辑这条消息，实时展示生成进度。等Agent生成完，预览消息就变成了完整的答案。

配置方法（以Discord为例）：
```json5
{
  channels: {
    discord: {
      streaming: {  // 开启流式预览
        blockStreamingCoalesceDefaults: { /* ... */ }
      }
    }
  }
}
```

需要注意的是，飞书的streaming实现方式不同——它通过**交互式卡片**来实现流式回复效果（`streaming`为布尔类型配置）。另外，如果平台对消息编辑有次数限制，或者不支持编辑，系统就不会发送流式预览，而是等待完整回复后再发送。

这就是**优雅降级**——有能力的平台提供更好的体验，没能力的平台至少保证功能可用。

### 3.3 平台特有的"超能力"

每个平台都有自己的杀手锏，翻译官会想办法利用这些特性：

**Discord的交互组件**

Agent返回的是普通Markdown：
```markdown
## 分析结果
- 发现问题：3个
- 建议修改：见下文
```

Discord翻译官可以把它转换成精美的交互式组件（Discord Components v2）：
```
┌─────────────────────────┐
│ 📊 分析结果              │
├─────────────────────────┤
│ 发现问题：3个            │
│ 建议修改：见下文         │
└─────────────────────────┘
```

> **注意**：Discord现在主要使用Components v2来构建交互界面，传统的Embeds虽然仍可用，但在有Components v2的情况下会被忽略。

**飞书的交互式卡片**

Agent说："请确认是否部署到生产环境"

飞书翻译官可以发送一个带"确认"和"取消"按钮的交互式卡片，用户点击按钮即可完成操作。飞书支持丰富的卡片模板和交互组件。

**消息编辑能力**

Agent生成了一段代码，后来发现少写了一个分号。Telegram翻译官可以直接编辑原消息，修正错误（通过`editMessageText`等API）。不同平台对消息编辑的支持程度不同，但主流平台（Discord、Telegram、飞书等）基本都支持消息编辑功能。

这些差异不是缺点，而是**特点**。好的翻译官会让每个平台都发挥出自己的特长。

## 4. 媒体文件：不只是"发个图"

当你让Agent"截图看看"或"发给我这个PDF"时，背后有一整套流程。

### 4.1 媒体的流转路径

```
用户："发给我项目结构图"
    ↓
Agent生成图片 → 保存到本地临时目录
    ↓
Gateway：消息对象 + 文件路径
    ↓
渠道适配器：从路径读取文件 → 上传到平台 → 用户看到图片
```

这里的关键是**媒体存储系统**。Agent生成的文件首先进入本地临时目录，翻译官发送时需要：

1. **从临时存储读取文件**：根据引用ID找到本地文件路径
2. **上传到平台**：每个平台有自己的上传API（Telegram使用`sendPhoto`等方法，Discord使用`send`方法配合file参数，等等）
3. **发送消息**：将媒体文件发送给用户（不同平台处理方式不同，有些直接上传，有些可能返回URL）

### 4.2 平台差异的处理

不同平台对媒体的处理差异很大。OpenClaw通过`mediaMaxMb`配置项来设置各平台的媒体大小限制，具体限制请参考各平台官方文档：

| 平台 | 配置方式 | 支持的格式 | 参考说明 |
|-----|-------------|-----------|---------|
| Telegram | 通过`mediaMaxMb`配置 | 常见图片格式（JPG、PNG、WEBP等） | 具体限制请参考Telegram Bot API官方文档 |
| Discord | 通过`mediaMaxMb`配置 | 全格式 | 具体限制请参考Discord官方文档 |


> **注意**：各平台的实际上传限制可能因账号类型、地区等因素有所不同，建议参考官方最新文档。

翻译官需要处理这些差异：
- 文件太大？先压缩，还是发网盘链接？
- 格式不支持？转换成通用格式？
- 上传失败？重试还是报错给用户？

## 5. 错误处理：当翻译官"掉线"了

多渠道架构带来一个复杂问题：**失败的处理**。网络抖动、平台API限流、Token过期——这些都是家常便饭。

### 5.1 错误分类：什么能重试，什么不能

OpenClaw针对不同错误类型有不同的处理策略。当翻译官发送失败时，系统会根据错误类型决定如何处理：

| 错误类型 | 说明 | 处理策略 |
|---------|------|---------|
| 渠道未启用 | 渠道配置为disabled或未启动 | 不重试，直接报错 |
| 速率限制（Rate Limit） | 收到429错误 | 等待后重试 |
| 临时错误 | 网络抖动、连接超时 | 重试几次 |
| 发送失败 | 内容违规、权限不足 | 不重试，报错 |

比如Discord翻译官发送消息时遇到429错误（Too Many Requests），系统就知道："哦，被限流了，等几秒再试"。

如果错误是"渠道未启用"，系统就知道："渠道都关了，重试也没用"。

### 5.2 自动重试与退避

当遇到可重试的错误时，系统不会立即重试，而是采用**指数退避**策略——等待时间随着重试次数增加而指数增长（例如：300ms、600ms、1200ms、2400ms...）。

这样既能避免"疯狂重试"把平台API打挂，又能自动恢复临时故障。具体的重试次数和间隔可以在配置中调整。

### 5.3 长消息自动分割

不同平台对单条消息的长度限制不同。OpenClaw通过`textChunkLimit`配置来设置各平台的消息长度限制：
- **Discord默认配置**：2000字符（与平台基础限制一致）
- **Telegram默认配置**：4000字符（官方限制4096字符，留有余量）

如果Agent生成的回复超长（比如一份详细的技术报告），OpenClaw会根据`textChunkLimit`配置自动切分发送。

但这里有个细节：**尽量不在代码块中间切断**。想象一下，如果一段代码被切成两半，可读性会很差。聪明的翻译官会找合适的分割点（比如段落边界、空行）。

## 6. 跨渠道的记忆：怎么知道"都是你"

最后来看一个关键问题：**跨渠道身份识别**。

假设你在Telegram上和Agent讨论项目，然后切换到Slack继续聊。Agent怎么知道这是同一个人？

### 6.1 规范化的用户ID

每个平台都有自己的用户ID格式：
- Telegram：纯数字或数字字符串，如 `123456789`（具体长度限制请参考Telegram官方Bot API文档）
- Discord：数字Snowflake ID，如 `123456789012345678`（API表现为字符串，通常为17-20位数字）
- 飞书：主要有三种格式：
  - **Open ID**（`ou_`开头）：应用内唯一，如 `ou_7dab8a3d3cdcc9da365777c7ad535d62`
  - **User ID**：租户内唯一，如 `3e3cf96b`
  - **Union ID**：开发商级别唯一（具体格式请参考飞书开放平台文档）

OpenClaw通过**identityLinks**配置实现跨渠道身份映射，使用平台前缀ID来识别用户：
```json5
{
  session: {
    identityLinks: {
      "zhangsan": ["telegram:123456789", "discord:123456789012345678", "slack:U12345"]
    }
  }
}
```

平台前缀ID采用`provider:originalId`的格式。这样，无论用户从哪个平台来，Agent都能识别出是同一个人。

### 6.2 共享的偏好 vs 隔离的上下文

这里要区分两个概念：

**共享的记忆（全局）**：
- 记忆工具中的用户偏好（"我喜欢用Tab缩进"）
- 项目事实（"项目用pnpm"）

**隔离的上下文（会话级别）**：
- 当前对话的历史
- 任务执行状态

举个例子：
- 你在Telegram私聊里让Agent"分析一下node_modules为什么这么肥大"——这是会话A
- 你在Discord频道里问"今天天气怎么样"——这是会话B

两个会话的上下文是隔离的，Agent不会在Discord里提node_modules的事。但如果你问"帮我写代码"，Agent在两边都会记得"你喜欢用Tab缩进"——因为这是全局偏好。

这就像你的助理有两个笔记本：一本记你的习惯（全局），几本记不同场合的对话（会话级别）。

## 7. 小结

这一章我们深入了多渠道接入的核心细节：

**核心设计**：
1. **Gateway架构**：渠道适配器通过实现`ChannelPlugin`接口接入，与Agent核心解耦
2. **标准消息格式**：统一的消息结构（入站`WebInboundMessage`、出站`ChannelOutboundContext`）让渠道适配规范化
3. **平台能力配置**：通过配置项启用各平台特长（交互组件、卡片、流式预览等）
4. **优雅降级**：平台能力不同时，自动调整策略保证体验
5. **错误处理**：根据错误类型（速率限制、临时错误等）采取不同处理策略
6. **跨渠道身份识别**：通过`identityLinks`配置实现多平台用户统一识别

**关键洞察**：
多渠道架构的最高境界是**"统一而不单调，多样而不混乱"**。核心逻辑写一次，通过适配器发挥每个平台的特长。这不是简单的"兼容所有平台"，而是"在每个平台上都能提供最好的体验"。
